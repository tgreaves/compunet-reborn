# Partyline

Partyline is Compunet's multi-user chat system.

## User Experience

### Entering and Exiting

- User connects via a BUY link on the Compunet system. This triggers the client download which is then executed.
- When the user exits Partyline, they are returned back to the main Compunet system.
- On entry, the message `{USERNAME} has entered partyline` is displayed and broadcast to all other users in Partyline. `{USERNAME}` is their Compunet user ID.
- Users start in the room: **lobby**.

### Screen Layout

- **Top area (blue border):** Scrolling message area. All text here is generated by the server.
- **Bottom area (green border):** User input area for text and commands. Has a green left-pointing cursor.

### Messages

When a message is received, the format is:

```
{USERNAME}:
Hello, this is my message.
```

The username and message are on separate lines. A message can span multiple lines.

### Sending Messages

- The user types in the bottom window.
- The message is transmitted when they press **RETURN twice**.

### Scrollback

- Use cursor keys **up/down** to scroll back through messages.

## Commands

Commands are prefixed with `*` and must be in lowercase. The user must press RETURN twice after each command.

| Command | Description |
|---------|-------------|
| `*help` | Display help text |
| `*alias {name}` | Set a friendly alias (replaces username in messages) |
| `*who` | List users currently in Partyline |
| `*enter {room}` | Enter a room (creates it if it doesn't exist) |
| `*call {username}` | Send a call notification to another user |
| `*dice {number}` | Roll a dice with the given number of sides |
| `*quit` | Exit Partyline and return to Compunet |

### *help

Displays:

```
Partyline help:-

Press RETURN twice to send commands
or messages.

Sample Partyline commands:-

*alias (followed by a name)
*who  (tells who's in pline)
*enter (any room name) to enter a different room
Use the cursor keys up/down to scroll back through messages
```

### *alias

Sets a friendly display name. This replaces the username when messages are displayed to other users. If no alias is set, the Compunet user ID is used.

### *who

Displays all users currently in Partyline:

```
Users in partyline:-

 {alias}.    ({compunet id})    {room}
```

If a user does not have an alias, it defaults to their Compunet ID.

### *enter

Enters a named room (e.g. `*enter pub`). If the room does not already exist, it is created. Messages can only be seen by other users in the same room.

When a user enters a room, others in that room see:

```
{USERNAME} has entered the room.
```

When a user leaves a room (by entering another), others in the old room see:

```
{USERNAME} has left the room.
```

### *call

Sends a notification to another user:

```
{sending_user} calls you from {room_name}
```

### *dice

Rolls a random number. `*dice 6` rolls a six-sided die.

Displayed to the user:

```
You have thrown x/6
```

Broadcast to the room:

```
{USERNAME} has thrown x/6
```

Where `x` is a random number from 1 to the specified number.

### *quit

Exits Partyline and returns to the Compunet terminal. Broadcasts: `{USERNAME} has left partyline`.

## Implementation Notes

### Architecture

- **Server module:** `server/partyline.py` — standalone module handling all partyline logic (rooms, users, message routing, command parsing). Keeps the main `compunet_server.py` clean.
- **Client program:** `client/c64/src/partyline/partyline.s` — 6502 program downloaded to C64 on demand. Handles screen rendering, keyboard input, and ACIA communication.

### Communication Protocol

After the partyline client is downloaded and executing, the session switches from X.25 framed protocol to a **raw line-based protocol**:

- Server marks the session as "partyline mode" — stops expecting X.25 packets.
- Both sides communicate via **CR-terminated ($0D) text lines**.
- No framing, no checksums, no sequence numbers.

**Server → Client (push model):**
- Server sends text lines as events occur (messages, notifications, command responses).
- The C64's NMI handler stores incoming ACIA bytes in the ring buffer at $CE00.
- The partyline client polls the ring buffer and displays received lines in the chat area.

**Client → Server:**
- Client sends completed input as raw bytes via ACIA TX register, CR-terminated.
- Transmitted after the user presses RETURN twice.

**Exit sequence:**
1. Client sends `*quit\r`
2. Server broadcasts leave message, sends `*EXIT\r` sentinel to client
3. Client sees `*EXIT\r`, restores screen, does RTS to return to terminal
4. Server switches session back to X.25 mode

### Server Module Design (`server/partyline.py`)

```python
async def handle_session(reader, writer, user_id):
    """Main partyline loop for one user. Returns when user quits."""
    # Manage: rooms, aliases, message routing
    # Blocks until *quit received or disconnect
```

Key responsibilities:
- **User state:** alias, current room, connected writer
- **Room management:** create on demand, track members, broadcast within room
- **Command dispatch:** parse `*` prefixed commands, execute, respond
- **Message broadcast:** send to all users in the same room
- **Async I/O:** use asyncio to handle concurrent reads/writes across multiple users

### C64 Client Design

The partyline client at $2000 handles:
- **Screen rendering:** direct writes to screen RAM ($0400) and colour RAM ($D800)
- **Chat area scrolling:** shift screen rows up when new messages arrive
- **Input handling:** keyboard polling via KERNAL GETIN, buffer until double-RETURN
- **ACIA receive:** poll NMI ring buffer ($CE00), display complete lines in chat area
- **ACIA transmit:** send bytes via ACIA data register when input complete

### Memory Map (during partyline)

| Address | Usage |
|---------|-------|
| $0400-$07E7 | Screen RAM (40x25) |
| $2000-$3FFF | Partyline client code + data |
| $CE00-$CEFF | NMI ring buffer (incoming ACIA bytes) |
| $D800-$DBE7 | Colour RAM |
| $D020-$D021 | VIC border/background |
| $DE00-$DE03 | ACIA registers (SwiftLink) |
